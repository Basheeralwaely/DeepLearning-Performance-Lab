Module docstring: Explain torch.compile as PyTorch 2.x's compilation path (replacing TorchScript for most use cases), what it does (graph capture via TorchDynamo, optimization via TorchInductor), and that this tutorial focuses on DIAGNOSTICS: understanding compilation overhead, mode tradeoffs, and graph break detection -- not production deployment (Phase 5).

Imports follow reference pattern plus:

```python
import torch._dynamo as dynamo
```

`main()` with these sections:

**Section 1: Setup** -- Banner, device, print_device_info(). Create SimpleCNN().to(device), get_sample_batch(). Log model info.

**Section 2: Eager vs Compiled Baseline** -- Measure eager (uncompiled) model with `@benchmark` decorator over NUM_TIMED forward passes (with `torch.inference_mode()`). Run NUM_WARMUP passes first for CUDA warm-up. Store as baseline result.

**Section 3: Compilation Overhead Measurement** -- For each mode in COMPILE_MODES:
- `dynamo.reset()` to clear compilation cache.
- `compiled = torch.compile(model, mode=mode)`
- Measure compilation time: time the FIRST forward pass with `time.perf_counter()` (this includes compilation). Log it explicitly as "Compilation time for {mode}: {t:.2f}s".
- Then run NUM_WARMUP additional warm-up passes.
- Measure runtime: use `@benchmark` over NUM_TIMED passes with `torch.inference_mode()`.
- Collect result dict: `{"name": f"compile({mode})", "time_seconds": ..., "memory_mb": ...}`.
- After all modes, include eager baseline in results list (name="eager") and call `print_benchmark_table(results)`.
- Log speedup of each compiled mode vs eager.

**Section 4: Graph Break Detection** -- Call `dynamo.reset()`. Run `explanation = dynamo.explain(model)(x)`. Log:
- `explanation.graph_count`
- `explanation.graph_break_count`
- `explanation.ops_per_graph`
- If break_reasons exist, log each with `reason.reason.split(chr(10))[0]` (first line only).
- If no breaks: log "Model is fully traceable -- no graph breaks detected".
- Log explanation: graph breaks force recompilation and reduce optimization opportunities.

**Section 5: Demonstrating Graph Breaks** -- Create a model with intentional graph breaks (a custom nn.Module that uses Python control flow like `if x.sum() > 0`). Run dynamo.explain on it. Show that this model HAS graph breaks. Compare its compile speedup vs the clean SimpleCNN. This teaches users to recognize and diagnose graph breaks.

Define the problematic model inline:
```python
class ModelWithGraphBreaks(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear1 = nn.Linear(3 * INPUT_SIZE * INPUT_SIZE, 256)
        self.linear2 = nn.Linear(256, 10)
        self.linear3 = nn.Linear(256, 10)

    def forward(self, x):
        x = x.view(x.size(0), -1)
        x = torch.relu(self.linear1(x))
        if x.sum() > 0:  # Data-dependent control flow = graph break
            return self.linear2(x)
        else:
            return self.linear3(x)
```

**Section 6: Key Takeaways** -- Log: compilation is a one-time cost (amortized over training), mode tradeoffs (default=fast compile, reduce-overhead=less kernel launch overhead, max-autotune=best runtime but slow compile), graph breaks reduce optimization, how to check for graph breaks before deploying compiled models, torch.compile is diagnostic-first (understand before optimizing).
