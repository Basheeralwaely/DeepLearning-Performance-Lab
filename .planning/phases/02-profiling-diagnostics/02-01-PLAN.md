`main()` function with these sections:

**Section 1: Setup** -- Print banner, call get_device()/print_device_info(), create SimpleCNN().to(device), get_sample_batch(), create nn.CrossEntropyLoss and torch.optim.SGD. Log model param count and batch info.

**Section 2: Profile a Training Step** -- Use `torch.profiler.profile` with:
- `activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA]`
- `schedule=schedule(wait=PROFILER_WAIT, warmup=PROFILER_WARMUP, active=PROFILER_ACTIVE, repeat=PROFILER_REPEAT)`
- `on_trace_ready=lambda p: p.export_chrome_trace(os.path.join("profiler_output", "training_trace.json"))`
- `record_shapes=True, profile_memory=True, with_stack=True`
- Create `profiler_output/` directory with `os.makedirs("profiler_output", exist_ok=True)` before the profiler context.
- Loop for `(PROFILER_WAIT + PROFILER_WARMUP + PROFILER_ACTIVE) * PROFILER_REPEAT` steps = 5 steps.
- Each step: `with record_function("training_step"):` forward, loss, backward, optimizer.step(), optimizer.zero_grad(). Call `prof.step()` after each iteration.

**Section 3: Analyze Console Output** -- Print three tables:
1. `prof.key_averages().table(sort_by="cuda_time_total", row_limit=15)` with header "Top Operations by GPU Time"
2. `prof.key_averages().table(sort_by="self_cpu_memory_usage", row_limit=10)` with header "Top Operations by CPU Memory"
3. `prof.key_averages(group_by_input_shape=True).table(sort_by="cuda_time_total", row_limit=10)` with header "Top Operations by GPU Time (grouped by input shape)"
- Log explanation of what each table shows and how to read it.

**Section 4: Profile Inference vs Training** -- Use `@benchmark` decorator to time a training loop (10 steps with grad) and an inference loop (10 steps with `torch.inference_mode()`). Use `print_benchmark_table` to show comparison. Each result dict must have keys "name", "time_seconds", "memory_mb".

**Section 5: Chrome Trace Instructions** -- Print instructions: "Open chrome://tracing in Chrome browser, click 'Load' and select profiler_output/training_trace.json". Log the file path.

**Section 6: Key Takeaways** -- Log: profiler schedule phases explained, which sort_by options to use when, inference_mode vs training memory differences, and when to use Chrome trace vs console tables.

**Create `profiling/memory_profiling.py`** (per D-01, D-02, D-05).

Module docstring: Explain GPU memory lifecycle during training (model params, activations, gradients, optimizer state), when to use memory profiling (OOM debugging, batch size optimization, memory leak detection), and what this tutorial demonstrates.

Constants:
```python
SMALL_BATCH = 32
SMALL_INPUT_SIZE = 32
LARGE_INPUT_SIZE = 224
NUM_CLASSES = 10
```

Imports follow reference pattern plus:
```python
from torch.utils.checkpoint import checkpoint
```

Define helper function `log_memory_state(label)` that logs:
- `torch.cuda.memory_allocated() / 1024**2` (Allocated MB)
- `torch.cuda.memory_reserved() / 1024**2` (Reserved MB)
- `torch.cuda.max_memory_allocated() / 1024**2` (Peak MB)
Format: `logger.info(f"[{label}] Allocated: {alloc:.1f} MB | Reserved: {reserved:.1f} MB | Peak: {peak:.1f} MB")`

`main()` with these sections:

**Section 1: Setup** -- Banner, device, print_device_info(). Log total GPU memory with `torch.cuda.get_device_properties(device).total_mem / 1024**2`.

**Section 2: Memory Lifecycle of a Training Step** -- Use SimpleCNN(input_size=SMALL_INPUT_SIZE) with SMALL_BATCH.
- `torch.cuda.reset_peak_memory_stats()` and `torch.cuda.empty_cache()` before starting.
- `log_memory_state("Before model creation")`
- Create model on device. `log_memory_state("After model.to(device)")`
- Create input batch. `log_memory_state("After input allocation")`
- Forward pass. `log_memory_state("After forward pass (activations stored)")`
- Backward pass. `log_memory_state("After backward pass (gradients computed)")`
- Optimizer step. `log_memory_state("After optimizer.step()")`
- `optimizer.zero_grad()`. `log_memory_state("After zero_grad()")`
- Print `torch.cuda.memory_summary(abbreviated=True)` for the full table.

**Section 3: Memory Scaling with Batch Size** -- Use SimpleCNN(input_size=LARGE_INPUT_SIZE) per D-02. Sweep batch_sizes = [8, 16, 32, 64, 128]. For each, reset peak stats, run forward+backward, record peak memory. Collect results as list of dicts with "name" (f"batch={bs}"), "time_seconds", "memory_mb". Use `print_benchmark_table(results)`. Wrap each iteration in try/except `torch.cuda.OutOfMemoryError` -- if OOM, log warning with batch size that failed and call `torch.cuda.empty_cache()`, then break. Delete model/tensors and empty_cache between sizes.

**Section 4: Catching OOM Gracefully** -- Demonstrate the OOM recovery pattern. Create SimpleCNN(input_size=LARGE_INPUT_SIZE). Start with a batch size likely to OOM (batch_size=256 on RTX 2070 with 224x224 input). Wrap in try/except `torch.cuda.OutOfMemoryError`. In except: log warning about OOM, call `torch.cuda.empty_cache()`, log_memory_state("After OOM recovery"), then retry with half the batch size. Log success message with the reduced batch size.

**Section 5: Gradient Checkpointing Memory Impact** (per D-05 -- profiling demo only). Use SimpleCNN(input_size=LARGE_INPUT_SIZE) with batch_size=64.
- **Without checkpointing:** forward through full model, backward, record peak memory with `torch.cuda.max_memory_allocated()`. Reset between runs.
- **With checkpointing:** Split forward pass: `features_out = checkpoint(model.features, x, use_reentrant=False)`, then `features_out = features_out.view(features_out.size(0), -1)`, then `output = model.classifier(features_out)`. Backward, record peak memory.
- Use `compare_results()` to show memory savings side by side. Log explanation: checkpointing trades compute for memory by not storing intermediate activations.

**Section 6: Key Takeaways** -- Log: memory stages explained, batch size is the primary memory knob, OOM recovery pattern, gradient checkpointing tradeoff (memory vs compute), when to use memory_summary() vs manual tracking.
